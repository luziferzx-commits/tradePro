# GoldBot Governance v1.0

## Version History
- **v1.0 (2026-06-20)**: Initial Release (Rules 0-5)
- **v1.1 (2026-06-23)**: Added Institutional Audit Rules (Rules 6-12)
- **v1.2 (2026-06-24)**: Added Lifecycle & Versioning Strictness (Rules 13-18)
*Effective Date for v1.2: 2026-06-24*

This document defines the strict governance, promotion, and deployment lifecycle for GoldBot, a Quant Research Platform.

## Rule #0
> **"If a result cannot be reproduced, it does not exist."**
> (หากผลลัพธ์ไม่สามารถสร้างซ้ำได้ ให้ถือว่าผลลัพธ์นั้นไม่มีอยู่จริง)
Every backtest, model training, or optimization run MUST be 100% reproducible. If another researcher cannot clone the repo, run the script, and get the exact same metrics, the result is invalid.

## Rule #1: Data Sufficiency ≠ Time
The end of the Observation Window is dictated by the **Evidence Gate**, not time. Time is only a minimum constraint, not a promotion condition.

## Rule #2: No Goalpost Moving
Data Sufficiency Gates (e.g., Signals >= 100, Trades >= 30) are absolute. They cannot be lowered or modified during the observation phase.

## Rule #3: Statistical Reasons for Extension
Any extension of the Observation Phase must specify exact statistical shortfalls (e.g., "Signals = 62 (<100)").

## Rule #4: Validate System, Not Strategy
The Observation Phase evaluates infrastructure stability, deterministic execution, data integrity, and logging completeness. It does NOT evaluate profitability, PF, or Win Rate.

## Rule #5: The Evidence Package
Promotion to the next phase requires a complete, unbroken Evidence Package. Missing a single file means "Promotion Blocked".

## Rule #6: Separation of Claim and Evidence
Never use words like "Completed", "Implemented", "Finished", or "Passed" without evidence. Statuses must be:
- `PLANNED`: Documented but not yet built/verified.
- `IMPLEMENTED`: Built but pending verification.
- `VERIFIED`: Proven by an Evidence Package.
- `PROMOTED`: Officially integrated into the next phase.

## Rule #7: Evidence Metadata
Every generated evidence file (CSV, JSON, MD) must include metadata: `Generated by`, `Generated at (UTC)`, `Git Commit`, `Dataset Version`, `Model Version`, and `SHA256` hash.

## Rule #8: Evidence is Immutable
Evidence packages must be stored in timestamped, immutable directories (e.g., `reports/baseline_YYYY-MM-DD/`). Existing baselines must never be overwritten.

## Rule #10: Negative Results Must Be Preserved
Failed experiments (e.g., negative PF, high drift, replay mismatch) must never be deleted. They must be recorded in the Experiment Registry just like successful ones, because knowing what DOES NOT work is as valuable as knowing what DOES.

## Rule #11: Every Promotion Must Reference Evidence IDs
When promoting a model or feature, the approval must explicitly reference the Evidence IDs (e.g., `Baseline-YYYYMMDD`, `WF-YYYYMMDD`). Traceability back to the exact evidence package is mandatory.

## Rule #12: Reproducibility Test Before Promotion
Before any promotion, the core pipelines (Train, Walk-Forward, Robustness) must be re-run. The results must match the original evidence within acceptable deterministic bounds (e.g., PF difference <= ±0.02, Trade Count <= ±2%). If results are non-reproducible, promotion is BLOCKED.

## Rule #13: Specification and Runtime Must Be Reviewed Separately
All components must undergo a two-layer review:
1. **Documentation Review (Layer 1)**: Verifies that the specification (e.g., ADR, Governance, Policy) is logically correct and approved.
2. **Runtime Review (Layer 2)**: Verifies that empirical data (Logs, SQLite, Metrics) actually reflects the documentation. If the system fails to manifest the documented behavior, it is marked `Specification: VERIFIED, Runtime: FAILED`.

## Rule #14: Documentation is Not Runtime Evidence
Documentation (e.g., `GOVERNANCE.md`) proves intent. Empirical outputs (e.g., `baseline_audit.md`) prove capability. They have equal importance but answer different questions. One cannot substitute the other.

## Rule #15: Evidence Must Be Actionable
Evidence must map to a clear decision matrix. Standard statuses are:
- `PASS`: Proceed normally.
- `PASS WITH OBSERVATION`: Proceed but monitor closely (e.g., minor drift).
- `INVESTIGATE`: Halt promotion to analyze underlying issues.
- `FAIL`: Reject current pipeline run.
- `BLOCK`: Hard stop due to critical failure (e.g., hash mismatch, data corruption).

## Rule #16: Every Block Requires Exit Criteria
If the system is `BLOCKED`, the exact conditions required to remove the block MUST be specified. A block without exit criteria is a dead end and is prohibited.

## Rule #17: Governance Changes Require Versioning
Every modification to `GOVERNANCE.md` must be tracked via a Version History block at the top of the file. It must include the Version Number, a Change Log, and the Effective Date.

## Rule #18: Governance Must Never Be Retroactive
A rule introduced in v1.2 (e.g., Effective Date: 2026-06-24) CANNOT be used to invalidate or re-litigate an Experiment that occurred on 2026-06-20. Older experiments are evaluated against the Governance version active at their time of execution.

## Rule #19: Governance Freeze During Observation
During an Active Observation Window, the following components are strictly FROZEN: Trading Logic, ML Models, Risk Parameters, Governance Rules, and Documentation. Changes are only permitted to resolve a "Critical Defect" (e.g., Hash Algorithm bug, Database Integrity failure) that actively prevents observation.

## Rule #20: Every Observation Window Ends With One Decision
Observation phases cannot end with ambiguity (e.g., "Continue observing"). Every window must conclude with one of four explicit verdicts:
- `PROMOTE`: Evidence passes all criteria.
- `EXTEND`: Insufficient sample size (must include reason).
- `INVESTIGATE`: Statistical anomalies detected.
- `TERMINATE`: Strategy proven unviable with no edge.

## The Golden Rule
> **"No metric, no merge."**
If there is no statistical data backing a change:
- DO NOT Merge
- DO NOT Promote
- DO NOT Deploy
Every change must be scientifically comparable and proven by data.

## Release Policy & Promotion Lifecycle
All models and features MUST follow this strict sequence. **Skipping steps is strictly prohibited.**
1. **Experimental** (Local prototyping)
2. **Research** (Backtest, Walk-Forward, Monte Carlo)
3. **Statistical Validation** (Out-of-sample confirmation)
4. **Shadow Trading** (Live market data, no real execution)
5. **Baseline Audit** (Post-shadow verification)
6. **Parallel Verification** (For architecture changes)
7. **Demo** (Forward testing with paper money)
8. **Production** (Live execution)

## Definition of Done (DoD) & Reproducibility Checklist
A feature or experiment is only considered "Done" if it passes all requirements below:
- [ ] Unit & Integration Tests Passed
- [ ] Backtest, Walk-Forward, Monte Carlo logs saved
- [ ] Dataset Version Hash & Random Seed documented
- [ ] Python Environment (Requirements) locked
- [ ] Model Hash saved
- [ ] Shadow Validation completed
- [ ] Artifacts (CSV/Report) generated
- [ ] Documentation updated

## Experiment Registry
Instead of vague references (e.g., "V3.2 tests"), all quantitative tests must have a formal Experiment ID (`EXP-YYYY-XXX`). Each experiment must record:
- Dataset Version
- Model Version
- Hyperparameters
- Git Commit Hash
- Result & Verdict
